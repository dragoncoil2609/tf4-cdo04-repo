0. Executive Summary

Hệ thống TF4 CDO-04 đóng vai trò là một SLO Early-Warning Control Plane with TSDB-backed Prediction Workflow dành cho 3 dịch vụ tier-1 cốt lõi trong nền tảng fintech demo. Mục tiêu thiết kế cốt lõi không chỉ là xây dựng dashboard giám sát thông thường, mà là chuyển hóa toàn bộ telemetry thô thành các quyết định dự báo (prediction decisions) có khả năng kiểm soát chi phí (cost guard), dự phòng (fallback), kiểm toán (audit), và cung cấp bằng chứng rõ ràng (evidence).

Hệ thống tiếp nhận telemetry qua POST /v1/ingest (được bảo vệ bởi API Gateway AWS_IAM/SigV4 service execute-api, sử dụng tenant token qua header X-Tenant-Ingest-Token), xuất metrics định dạng Prometheus tại route nội bộ /metrics để ADOT Collector sidecar thực hiện scrape rồi đẩy lên Amazon Managed Service for Prometheus (AMP) bằng phương thức remote_write có ký SigV4. Định kỳ mỗi 5 phút, EventBridge Scheduler sẽ gửi các job dự báo vào SQS. Prediction Worker thực hiện long-poll SQS, truy vấn AMP lấy dữ liệu cửa sổ lookback 120 phút, đồng bộ và làm sạch thành các bucket 1 phút, sau đó gọi AI Engine thông qua API Gateway HTTP API (được ký AWS_IAM → VPC Link → internal ALB listener cổng :80 → AI target group). Kết quả dự báo được lưu trữ kiểm toán vào DynamoDB và phát cảnh báo qua SNS nếu phát hiện rủi ro cao. Cơ chế ECS Service Connect được giữ lại làm phương án rollback/fallback trong quá trình di trú (migration). Khi AI gặp sự cố hoặc xuất hiện khoảng trống dữ liệu lớn, worker tự động chuyển đổi sang static threshold fallback (ngưỡng tĩnh).

Thiết kế hiện tại vận hành trên vùng us-east-1, sử dụng dịch vụ ECS Fargate Linux/x86 trong các subnet riêng tư (private subnets). API Gateway HTTP API đóng vai trò là cổng công cộng (public front door) duy nhất với 3 route: GET /health (NONE auth), POST /v1/ingest (AWS_IAM/SigV4 + X-Tenant-Ingest-Token), và POST /v1/predict (AWS_IAM/SigV4). API Gateway kết nối qua VPC Link v2 tới các ENIs nằm trong các private subnets, chuyển tiếp dữ liệu đến internal ALB HTTP cổng :80 → các target groups cổng :8080 của ECS. Các route /health và /v1/ingest được ALB định tuyến đến target group telemetry-api; route /v1/predict được định tuyến đến target group ai-engine. ALB nằm hoàn toàn trong private subnets, không có cổng Ingest công cộng 80/443, cũng như không sử dụng cổng :8443 cũ. Hạ tầng sử dụng 1 NAT Gateway, S3/DynamoDB Gateway Endpoints, AMP làm TSDB, DynamoDB làm audit/policy store, S3 làm evidence/failure-buffer/baseline store, CloudWatch cho alarms/dashboard/logs, và Budgets kết hợp Lambda làm cost breaker. ACM cert được giữ lại phục vụ cho custom domain của API Gateway trong tương lai; hiện tại chưa cấu hình tài nguyên API Gateway custom domain. CI/smoke test thực hiện cuộc gọi trực tiếp qua api_gateway_base_url (execute-api endpoint).

Lưu ý quan trọng về thiết kế Scrape: Không có Prometheus server scrape trực tiếp 3 dịch vụ tier-1. Đây là kiến trúc push telemetry → Telemetry API /metrics → ADOT scrape → AMP remote_write. Vì vậy, kết quả thử nghiệm 50 RPS trong k6 chỉ là giới hạn chịu tải của Telemetry Ingest API, không phải là bằng chứng AMP đã lưu trữ thành công 50 mẫu/giây.

3. Runtime Flow Chi Tiết

3.1 Luồng Thu Nhận Telemetry (Telemetry Ingest Flow)

Ứng dụng Telemetry API chạy tại src/telemetry_api/main.py (chạy lệnh uvicorn telemetry_api.main:app --port 8080).

1
Client gửi Request
Client gọi POST /v1/ingest qua API Gateway HTTP API (yêu cầu ký AWS_IAM/SigV4 và đính kèm token tại header X-Tenant-Ingest-Token).
2
Định Tuyến Qua VPC Link
API Gateway chuyển tiếp qua VPC Link v2 → ENIs trong private subnets → internal ALB HTTP :80 → Telemetry target group :8080.
3
Middleware Xử Lý
CorrelationIdMiddleware gán request ID; PayloadSizeLimitMiddleware từ chối payload lớn hơn 65,536 bytes.
4
Schema & Tenant Verification
Yêu cầu header X-Tenant-Id và phải khớp với body tenant_id. Schema được validate qua Pydantic TelemetryPayload.
5
PII & Cardinality Filter
Từ chối các metric không nằm trong 7 AI signals. Loại bỏ hoàn toàn PII hoặc nhãn có cardinality cao (email, phone, transaction_id, user_id, request_id, trace_id, prediction_id).
6
Observe Metric
PrometheusExporter cập nhật gauge trong bộ nhớ, sẵn sàng để scrape qua route nội bộ GET /metrics. Trả về mã 201/202.

Cấu hình AWS Mode: APP_MODE=aws, TELEMETRY_STORAGE_BACKEND=prometheus_amp, và AMP_DELIVERY_ENABLED=false. Ứng dụng không ghi trực tiếp lên AMP mà xuất tại `/metrics` để ADOT sidecar scrape.

3.2 Luồng Scrape & AMP Remote Write
1
ADOT Collector Scrape
ADOT Collector sidecar scrape endpoint localhost:8080/metrics mỗi 15 giây.
2
Ký Chữ Ký SigV4
Sử dụng exporter prometheusremotewrite và ký SigV4 cho vùng us-east-1 (service aps).
3
Ghi Lên AMP Workspace
Đẩy dữ liệu time-series lên Amazon Managed Service for Prometheus (AMP).

Trade-off Single-Writer: Số lượng task mong muốn (desired count) của Telemetry API giảm từ 2 xuống 1, task size tăng lên 1024 CPU / 2048 MB. Điều này tránh xung đột ghi/đọc và tối ưu chi phí trong môi trường Sandbox.

3.3 Luồng EventBridge Scheduler → SQS Flow

EventBridge Scheduler kích hoạt 3 job (cho 3 dịch vụ tier-1) mỗi 5 phút đẩy vào SQS queue:

// SQS Message Payload { "tenant_id": "demo-tenant-001", "service_id": "payment-gw", // payment-gw | ledger | fraud-detector "lookback_window_minutes": 120, "prediction_mode": "balanced" 3.4 Luồng Xử Lý Của Prediction Worker

Chạy tại src/prediction_worker/app.py (outbound-only). Luồng hoạt động chi tiết:

1
SQS Long Polling
Worker gọi receive_message với thời gian chờ 20 giây (WaitTimeSeconds=20).
2
Truy Vấn AMP
Thực hiện 7 cuộc gọi PromQL query_range tuần tự lấy dữ liệu lookback 120 phút (step=60s). Sử dụng SigV4 để ký xác thực.
3
Đồng Bộ & Làm Sạch (Align & Impute)
Align dữ liệu thành các 1-minute buckets. Thực hiện forward-fill cho hầu hết các tín hiệu, riêng queue_depth sử dụng zero-fill.
4
Kiểm Tra Khoảng Trống Dữ Liệu
Nếu tỷ lệ trống dữ liệu gap_ratio >= 0.5 hoặc dữ liệu AMP trống, tự động bỏ qua cuộc gọi AI và chuyển hướng sang Static Threshold Fallback.
5
Gọi Predict API của AI Engine
Thực hiện cuộc gọi SigV4 POST /v1/predict đến API Gateway execute-api url (timeout 2s). Nếu AI lỗi/timeout, kích hoạt static fallback.
6
Ghi Nhận Kiểm Toán & Alert
Ghi kết quả vào DynamoDB Audit table (cơ chế idempotency). Phát SNS alert nếu anomaly=true và severity >= 0.8. Xóa message SQS.
3.5 Luồng Xử Lý Của AI Engine

Chạy tại src/ai_engine/app/main.py (cổng 8080).

1
Rate Limit Check
RateLimitMiddleware giới hạn tối đa 600 req/phút/tenant, trả về 429 nếu vượt ngưỡng.
2
Tải Baseline & Trừ Mùa Vụ
Tải baseline từ S3. Trừ đi seasonal profile của STL (1440 điểm theo phút trong ngày) để lấy Residual. Nếu không có baseline, tự động fallback sang in-window z-score.
3
EWMA Control Chart
Sử dụng EWMA với alpha=0.3. Đánh giá vượt ngưỡng kiểm soát trên K = 4.0 * sigma * sqrt(alpha/(2-alpha)). Đề xuất hành động tương ứng.
4
Confidence Gating & Response
Nếu độ tin cậy confidence < 0.7, hạ cấp đề xuất xuống INVESTIGATE. Trả về kết quả JSON đầy đủ.
3.6 Tự Động Ngắt Chi Phí (Cost Breaker Flow)

Hệ thống cấu hình một Lambda Cost Breaker (Python 3.10) liên kết với SNS của AWS Budgets. Khi chi phí thực tế chạm mốc 100% ngân sách ($200/tháng), Lambda sẽ tự động scale down desiredCount của prediction-worker và ai-engine về 0 để dừng phát sinh chi phí, trong khi vẫn giữ telemetry-api hoạt động để bảo toàn ingest và evidence.

4.6 Deployment Contract cho AI Engine

Môi trường ECS Fargate, sử dụng 0.5 vCPU / 1GB memory. Task mong muốn chạy tối thiểu/desired 2 tasks, tối đa 4 tasks (Max). Nghe kiểm tra sức khỏe tại endpoint /health bằng thư viện urllib của Python (Chu kỳ 30 giây, healthy 2x200, unhealthy 3x non-200). Các baseline tĩnh được đồng bộ từ tiền tố baselines/ của S3 evidence bucket bảo vệ bằng khóa KMS.

Cơ chế Rollout và Rollback: Để tối ưu hạ tầng Sandbox, cơ chế CodeDeploy Canary được phân loại là Post-MVP. Hiện tại, hệ thống sử dụng ECS Rolling Deployment với Circuit Breaker và Auto Rollback (thiết lập enable = true và rollback = true tại ai_engine.tf). Khi Task Definition mới gặp sự cố khởi động (như lỗi crash-loop, sai port, hoặc health check fail), ECS tự động ngắt và thực hiện rollback tức thời về phiên bản Task Definition cũ hoạt động tốt gần nhất để đảm bảo an toàn.

5. Infrastructure as Code

5.1 Cấu Hình Terraform Backend

Yêu cầu Terraform >= 1.10.0 và AWS provider >= 5.80.0. Backend S3 native lockfile:

terraform { backend "s3" { bucket = "tf4-cdo04-terraform-state-0e0bped4" key = "tf4-cdo04/sandbox/terraform.tfstate" region = "us-east-1" encrypt = true use_lockfile = true } }
5.2 Bootstrap Hạ Tầng

Bootstrap tạo S3 state bucket với versioning, SSE-S3 encryption, public access block. Thêm bucket policy từ chối kết nối phi TLS (aws:SecureTransport=false). Sử dụng prevent_destroy=true cho S3 state bucket. Thiết lập OIDC provider cho GitHub và tạo role deploy tf4-cdo04-github-deploy-role (không dùng quyền AdministratorAccess toàn phần). Tạo user tin giới hạn quyền đọc logs, query AMP và S3 evidence, nhưng bị chặn truy cập state bucket.

5.3 Thiết Kế Mạng VPC và Nhóm Bảo Mật

VPC CIDR mặc định 10.0.0.0/16, chia thành 2 Public subnets và 2 Private subnets trải trên 2 AZs. Sử dụng duy nhất 1 NAT Gateway zonal nằm ở Public subnet[0] để tiết kiệm chi phí. Cấu hình S3 và DynamoDB Gateway Endpoints giúp định tuyến trực tiếp trong private subnets, không tốn phí NAT.

Security Group Ingress Rules Egress Rules
Internal ALB HTTP 80 từ VPC Link ENIs subnets (private subnets) TCP 8080 tới Telemetry API SG + AI Engine SG
Telemetry API TCP 8080 từ Internal ALB SG Mọi outbound
Prediction Worker Không có (Outbound-only) TCP 8080 tới AI Engine SG + Mọi outbound
AI Engine TCP 8080 từ Internal ALB SG + TCP 8080 từ Worker SG Mọi outbound
5.4 Mô-đun Dữ Liệu (Data Module)

Gồm AMP Workspace tf4-cdo04-<env>; DynamoDB audit table tf4-cdo04-audit-<env> (PAY_PER_REQUEST, TTL expires_at_epoch, GSI prediction-index); DynamoDB policy table tf4-cdo04-service-policies-<env> chứa seed policies cho ledger, payment-gw, fraud-detector (threshold 85.0); SQS queue tf4-cdo04-prediction-<env> (visibility 180s, DLQ maxReceiveCount=5, retention 14 ngày); S3 evidence bucket tf4-cdo04-evidence-<env> (xóa failure-buffer/ sau 7 ngày, các vật thể khác sau 90 ngày); KMS alias alias/tf4-cdo04-<env> (tự động xoay vòng); và Secrets Manager tf4-cdo04/<env>/tenant-ingest-token (tạo tự động bằng random_password, inject qua task definition).

5.5 Mô-đun Tính Toán (Compute Module)

ECS Fargate Cluster tf4-cdo04-<env>-cluster, Container Insights được kích hoạt. Tasks chạy trong private subnets với assignPublicIp=DISABLED. ECR repositories được cấu hình scan_on_push=true, tag mutability là IMMUTABLE, và có vòng đời giữ tối đa 10 tagged images, xóa ảnh untagged sau 14 ngày.

5.6 ALB và API Gateway

Internal ALB chạy scheme internal, nằm trong private subnets. Cấu hình HTTP :80 listener, có 2 rule định tuyến: priority 10 forward /health và /v1/ingest tới telemetry-api; priority 20 forward /v1/predict tới ai-engine. API Gateway HTTP API là cổng công cộng duy nhất, sử dụng VPC Link v2 kết nối tới các private ENIs của ALB. ACM certificate được tạo và xác thực bằng DNS cho domain xbrain26hackathon269.software để dùng cho API Gateway custom domain sau này.

7. Security Posture

7.1 Bảo Vệ An Ninh Mạng

Tất cả container ECS Fargate đều chạy trong Private Subnets, cấu hình assignPublicIp=DISABLED. Chỉ duy nhất API Gateway HTTP API được công khai ra internet. Internal ALB chạy ở chế độ internal (chỉ giao tiếp trong private subnet). Prediction Worker chỉ có kết nối Outbound, không mở cổng Ingress. AI Engine chỉ nhận kết nối từ Internal ALB SG và Worker SG trên cổng 8080. S3 và DynamoDB được định tuyến nội bộ qua VPC Gateway Endpoints, không tốn phí NAT.

7.2 Quản Lý Định Danh và Quyền Hạn Tối Thiểu (IAM)

CI/CD sử dụng GitHub OIDC với deploy role chuyên dụng tf4-cdo04-github-deploy-role thay thế AWS access key tĩnh. Quyền hạn của ECS Task được tối giản hóa cho từng service: Telemetry chỉ ghi SQS, write AMP, put S3 failure-buffer; Worker chỉ consume SQS, query AMP, write DynamoDB và SNS publish; AI Engine chỉ đọc baselines từ S3.

Rủi ro Deploy Policy: Một số tài nguyên AWS không hỗ trợ tag-on-create lúc khởi tạo, do đó chính sách GitHub OIDC deploy vẫn tồn tại các quyền wildcard rộng như ssm:* và cloudwatch:* trên *. SRE cần thực hiện review định kỳ.

7.3 Bảo Vệ Dữ Liệu & 7.4 Xác Thực API và Sự Lệch Pha Trạng Thái (Auth Mismatch)

Tất cả S3 bucket và DynamoDB table đều được mã hóa tĩnh (SSE). Telemetry API lọc bỏ dữ liệu PII tại ranh giới Ingest.

Đối với Ingest API: Xác thực SigV4 được cấu hình cứng tại API Gateway HTTP API qua AWS_IAM. Client/K6 ký yêu cầu SigV4 cho service execute-api và đính kèm token tại X-Tenant-Ingest-Token. Các yêu cầu không ký hoặc token không hợp lệ bị từ chối 403 tại API Gateway.

Lệch hợp đồng AI Auth (SYS-09): Kỹ sư vận hành cần lưu ý: Prediction Worker gửi yêu cầu gọi AI được ký SigV4 qua API Gateway HTTP API (AWS_IAM) ở Path A, tuy nhiên bản thân AI Engine App ở phía đối diện không thực thi bất kỳ cơ chế kiểm soát chữ ký SigV4 nào mà chỉ kiểm tra header X-Tenant-Id. Việc thực thi SigV4 app-side vẫn đang nằm trong danh mục hardening sản phẩm.

11. Stale Docs & Drift

11.1 Chi Tiết Các Điểm Lệch Pha Kiến Trúc (P0/P1 Mismatches)
Mã Lệch Pha Khu Vực Phát Sinh Mô Tả Chi Tiết Điểm Lệch Pha Mức Độ Ảnh Hưởng Hướng Khắc Phục Chỉ Định
SYS-01 Edge Security Tài liệu yêu cầu HTTPS cứng trên ALB; cấu hình thực tế sử dụng API Gateway HTTP API là public edge với internal ALB sau VPC Link. Rủi ro an toàn mạng ở mức biên. API Gateway HTTP API là public edge; ACM cert được giữ cho future custom domain; HTTPS enforced ở API Gateway khi custom domain gắn vào.
SYS-02 Ingress CIDR ALB từng được cấu hình public với 0.0.0.0/0; hiện tại ALB đã được chuyển thành internal, nằm trong private subnets. Accidental public exposure resolved. API Gateway HTTP API là sole public entry point; internal ALB chỉ nhận traffic từ VPC Link ENIs.
SYS-03 Terraform Backend Tài liệu cũ hướng dẫn chạy apply kèm cờ -lock=false, trong khi backend thực tế có hỗ trợ lockfile. Tranh chấp trạng thái gây hỏng file state. Loại bỏ hoàn toàn cờ -lock=false, tận dụng cơ chế s3 native lockfile.
SYS-08 AI Runtime Có nghi ngờ về việc AI Engine app chạy stub/mock placeholder. Nghi ngờ chất lượng evidence. Đã chứng minh qua logs thực tế có kết quả dự đoán với status 200 và complete_window.
SYS-09 AI Auth Layer Worker gửi thông tin xác thực SigV4 nhưng AI Engine app không hề cấu hình xác minh SigV4 ở tầng ứng dụng. Mất an toàn bảo mật nội bộ (P0). Chấp nhận API Gateway AWS_IAM là front door bảo vệ cho MVP; cấu hình SigV4 app-side là production hardening.
SYS-11 AMP Ingestion Trong code tồn tại file stub adapter ghi AMP nhưng thực tế luồng ghi là do ADOT Collector. Gây bối rối cho lập trình viên mới. Xóa/comment file stub, ghi chú cơ chế ghi duy nhất qua ADOT.
SYS-12 k6 Test Path Một số script kiểm thử cũ gửi request tới route /v1/telemetry cũ. Test script chạy lỗi 404. Cập nhật toàn bộ script test trỏ về route mới /v1/ingest.

Khi AI Engine gặp sự cố (timeout > 2s, lỗi 5xx, 429 hoặc lỗi mạng), Prediction Worker tự động kích hoạt chế độ Fallback tĩnh, ghi log audit với prediction_source=static_threshold_fallback và mô tả chi tiết lý do lỗi tại fallback_reason. Khi truy vấn AMP trống hoặc tỷ lệ trống gap_ratio >= 0.5, Worker tự động bỏ qua cuộc gọi AI Engine, chuyển sang dùng Static Threshold Fallback, và ghi nhận trạng thái dữ liệu là partial_window. SRE cần đọc logs ADOT Collector, kiểm tra active series của AMP, và xem logs của Telemetry API.

ADOT AWS Distro for OpenTelemetry — Sidecar Collector chịu trách nhiệm scrape dữ liệu và đẩy lên AMP. Đảm bảo việc phân tách luồng đẩy dữ liệu khỏi code ứng dụng chính.
AI Engine Dịch vụ chạy container thực hiện gọi phân tích bất thường dựa trên thuật toán STL và EWMA. Route chính là API Gateway HTTP API AWS_IAM -> VPC Link -> ALB. Thực thi phân tích thông minh và đề xuất thay đổi quy mô.
Prediction Worker Tiêu thụ job từ SQS, thực hiện truy vấn AMP, căn chỉnh mốc thời gian và gọi AI Engine. Bộ não điều khiển phối hợp quy trình dự báo của Control Plane.
Static threshold fallback Chế độ tự động chuyển đổi sang so sánh ngưỡng tĩnh từ DynamoDB khi AI Engine sập hoặc dữ liệu bị mất mát. Đảm bảo tính liên tục của hệ thống cảnh báo sớm.
Evidence status Trạng thái ghi nhận dữ liệu lookback trong audit log (complete_window hoặc partial_window). Bằng chứng chứng minh chất lượng dữ liệu đầu vào phục vụ dự báo.
Service Connect Mạng lưới giao tiếp dịch vụ nội bộ của ECS Fargate không qua cổng ALB công cộng. Dùng làm fallback/rollback. Tăng cường tính bảo mật kết nối nội bộ.
Cost breaker Hàm Lambda tự động scale-down Worker và AI Engine về 0 khi tổng chi phí chạm mốc $200 ngân sách tháng. Chốt chặn bảo vệ chi phí tài nguyên cloud.
